# 주식 애널리스트 리포트 모바일 웹앱 — 개발 계획서 (v2)

> 종목명을 입력하면 기업정보 · 재무제표 · 실적 · 밸류에이션 · 적정주가 · 목표주가를
> 증권사 애널리스트 보고서 수준으로 자동 생성하는 스마트폰 웹앱
>
> **v2 변경 요지**: ① 선행 EPS 의존 제거(후행 EPS 시나리오 + RIM 주력), ② LLM 출력에서
> 수치 필드 제거(서술 전용) + 프로그램적 수치 검증 추가, ③ 업종별 밸류에이션 분기 명시,
> ④ 비공식 라이브러리 → 공식 API 전환 계획, ⑤ MVP UI를 1열 카드 스크롤로 단순화,
> ⑥ 목표주가 3밴드(보수/기본/낙관) 제시, ⑦ 수직 슬라이스 실행 전략 명문화
>
> **v3 변경 요지**(§5.2): v2의 "선행 EPS 완전 배제"가 사이클 저점 종목을 과소평가하는
> 비현실적 결과를 낳아 개선한다. 사실(출처 명시 컨센서스)과 가정(규칙 상수)을 구분해,
> 네이버 금융 증권사 컨센서스(선행 EPS·목표주가)를 밸류에이션에 도입하고, 순환 구조였던
> "EPS×현재PER" 멀티플은 폐기한다. RIM(내재가치)과 컨센서스(선행)를 유형별로 가중평균한다.

## 1. 제품 정의

| 항목 | 내용 |
|---|---|
| 입력 | 종목명 또는 종목코드 (자동완성 지원) |
| 출력 | 애널리스트 보고서: 투자의견, 목표주가(3밴드), 기업개요, 재무분석, 실적 추이, 밸류에이션, 리스크 |
| 대상 시장 | 국내 상장기업 (KOSPI / KOSDAQ) — 1차 범위 |
| 플랫폼 | 모바일 우선 반응형 웹앱 (PWA로 홈화면 설치 가능) |

**핵심 차별점**: ① 수치 근거가 명시된 밸류에이션(적정주가 산출 로직과 가정 공개),
② LLM이 작성하는 자연어 투자 논거(Investment Thesis), ③ 공시 원문(DART) 기반의 팩트 인용.

**성패를 가르는 대원칙**: 애널리스트 보고서 퀄리티의 핵심은 LLM의 문장력이 아니라
**실제 데이터**다. 모든 수치는 코드가 계산하고, LLM은 계산된 수치를 해석·서술만 한다.
LLM 출력에는 수치·투자의견 필드를 아예 두지 않는다(§5.4).

## 2. 데이터 소스 및 API 연동 전략

### 2.1 DART 전자공시 (OpenDART, opendart.fss.or.kr)
- **발급**: 무료 API 키 (일 20,000건 한도) — Phase 0에서 발급
- **활용 API**:
  - `기업개황(company.json)` — 회사 기본 정보, 업종, 대표자
  - `단일회사 전체 재무제표(fnlttSinglAcntAll)` — 손익계산서/재무상태표/현금흐름표 (연간·분기, 연결·별도)
  - `공시검색(list.json)` — 최근 주요 공시 (실적발표, 유상증자, 자사주 등)
  - `배당에 관한 사항`, `최대주주 현황` — 주주환원·지배구조 분석
- **주의**: 종목코드 ↔ DART 고유번호(corp_code) 매핑 테이블 필요 (`corpCode.xml` 전체 다운로드 후 DB화)
- **계정과목 정규화 필수**: 계정명은 회사·보고서 유형(연결/별도)마다 다르다.
  핵심 계정 15개(매출액, 영업이익, 당기순이익, 자본총계, 부채총계 등)의 명시적 정규화
  매핑 테이블을 두고, 매핑 실패 시 조용히 0 처리하지 않고 오류로 드러낸다.

### 2.2 KRX 시세 데이터 — 2단계 전략
| 단계 | 소스 | 비고 |
|---|---|---|
| 프로토타입 (Phase 1~4) | `pykrx` (비공식 래퍼) | 빠른 개발. 단, KRX 웹 스크래핑 기반이라 사이트 개편 시 파손, 클라우드 IP 차단 가능성 있음 |
| 운영 (Phase 6) | **KRX 정보데이터시스템 공식 Open API** (data.krx.co.kr) | 배포 전 반드시 전환. 클라이언트 모듈을 인터페이스로 추상화해 교체 비용 최소화 |

- **활용 데이터**: 일별 시세(OHLCV), 시가총액, 상장주식수, 업종(섹터) 분류, 업종별 평균 PER/PBR/배당수익률

### 2.3 보조 데이터 소스
| 소스 | 용도 | 비고 |
|---|---|---|
| 한국투자증권 KIS Developers | 실시간/지연 시세 | 무료, OAuth. 선택 사항 |
| 네이버 검색 API | 최신 뉴스 헤드라인 | 리포트의 "최근 이슈" 섹션 |
| 네이버 금융 컨센서스 | 선행 EPS·목표주가 평균·투자의견 평균 | 모바일 통합 JSON(`m.stock.naver.com/api/stock/{code}/integration`). 비공식 엔드포인트지만 다수 증권사 컨센서스를 집계·공개. 선행 밸류에이션 보정에 사용 (§5.2 v3). 커버리지 없으면 null → RIM 단독 폴백 |

### 2.4 MCP 활용 방법
- **개발 단계 (Claude Code / claude.ai에서)**:
  - 공개 한국주식 MCP 서버(`korea-stock-mcp`, `dartpoint-mcp`, `pykrx-mcp` 등)를 등록해
    API 응답 스키마 탐색, **보고서 포맷 프로토타이핑(Phase 0의 핵심 산출물)**, 데이터 검증을 대화형으로 수행
  - 커뮤니티 MCP는 유지보수 여부를 먼저 확인하고, 프로토타이핑 용도로만 쓴다
- **운영 단계 (앱 백엔드에서)**:
  - 백엔드가 DART/KRX REST를 직접 호출하고, Claude API에는 **tool use**로 자체 도구를 정의해 전달
  - 커뮤니티 MCP 서버를 운영 데이터 경로에 넣지 않는다 (가용성·보안 통제 불가)
- **원칙**: MCP는 "개발·탐색 가속기", 운영 데이터 경로는 캐시를 갖춘 자체 데이터 레이어로 고정

## 3. 시스템 아키텍처

```
[모바일 브라우저 (PWA)]
        │ HTTPS
[Next.js 프론트엔드 + API Routes (Vercel)]
        │
        ├── [데이터 레이어] ── DART OpenAPI / KRX(pykrx→공식API) / (선택)KIS
        │         │
        │      [캐시: Upstash Redis]  ← TTL 차등: 재무제표=분기, 시세=일, 종목마스터=월
        │
        ├── [분석 엔진] ── 재무비율 · 밸류에이션 모델 (순수 계산 모듈 + 단위 테스트)
        │
        └── [리포트 생성기] ── Claude API (tool use 스키마 강제, 서술 전용)
                  │
              [수치 검증기] ← LLM 서술 속 숫자를 입력 JSON과 프로그램적으로 대조
```

- **프론트엔드**: Next.js(App Router) + Tailwind CSS, 차트는 lightweight-charts(주가) + Recharts(재무)
- **백엔드**: Next.js API Routes (서버리스) — API 키는 반드시 서버 측에만 보관, 클라이언트 노출 절대 금지
- **캐시 TTL 차등이 필수인 이유**: 재무제표(분기)와 시세(일)의 수명이 다르다. 단일 캐시로 묶으면
  어제 주가로 상승여력이 계산되는 버그가 생긴다
- **DART 한도 관리**: 일 20,000건 한도 대비 캐시 적중률을 모니터링 지표로 둔다

## 4. 화면 구성 (모바일 우선)

### 4.1 MVP — 세로 1열 카드 스크롤 (탭 없음)
단순함 우선. 위에서 아래로:

1. **헤더 카드** — 종목명 · 현재가 · 등락률 · 투자의견 배지(매수/중립/매도) · 목표주가(기본 밴드) · 상승여력%
2. **투자 요약** — 핵심 논거 3줄 (불릿)
3. **주가 차트** — 최근 1년 (1M/3M/1Y 토글)
4. **실적 테이블** — 매출/영업이익/순이익 3~5개년 + YoY 성장률
5. **밸류에이션 카드** — 목표주가 3밴드(보수/기본/낙관) + 산출 방법론·가정 명시, Peer 비교 테이블
6. **강점 · 리스크** — 2열 대비 카드
7. **애널리스트 코멘트** — 종합 서술
8. **면책 고지** — 하단 고정

- 색상 관례: **상승 빨강 / 하락 파랑** (국내 관례), 다크모드 대응
- 하단 고정 바: 현재가 · 등락률 상시 표시
- 로딩 UX: 생성에 10~30초 소요 → **단계별 진행 표시** ("재무 수집 중" → "밸류에이션 계산" → "보고서 작성")

### 4.2 확장 (Phase 6 이후)
- 검색 홈(자동완성, 최근 조회, 관심종목), 탭 분리(기업개요/재무분석 상세), PER/PBR 히스토리컬 밴드 차트, PDF 내보내기/공유

## 5. 밸류에이션 방법론 (분석 엔진 스펙)

### 5.1 업종별 방법론 분기 (필수)
단일 공식은 금융·지주·적자기업에서 반드시 깨진다. 시총 상위에 금융·지주가 많으므로 예외가 아니라 본편이다.

| 기업 유형 | 판별 기준 | 주력 방법론 |
|---|---|---|
| 일반 제조/서비스 | 기본값 | RIM(내재가치) + 증권사 컨센서스 가중평균 (§5.2 v3) |
| 금융 (은행·보험·증권) | KRX 업종분류 | **PBR–ROE 모델** (PER/EV-EBITDA 부적합) |
| 지주회사 | 회사명·업종 판별 | NAV 할인 (1차 범위에서는 PBR–ROE로 근사, 한계 명시) |
| 적자기업 (EPS ≤ 0) | 최근 연간 EPS | PSR 참고치 제시 또는 **"목표주가 미제시"** — 음수 EPS에 멀티플을 곱하지 않는다 |

### 5.2 미래 추정: 사실 vs 가정 (v3 개정)
> **v3 개정 배경**: v2는 "확보 가능한 공식 선행 EPS 소스가 없다"는 이유로 후행 실적만
> 썼다. 그러나 후행 실적이 사이클 저점에 있는 종목(예: 반도체)은 현저히 과소평가된다.
> 삼성전자 사례에서 후행 기준 목표주가(172,356원)와 증권사 컨센서스(513,958원)의 3배
> 괴리는 전적으로 선행 실적을 무시한 데서 나왔다. "현실에 맞지 않는 원칙은 개선한다."

수치 창작 금지 원칙은 **날조(데이터에 없는 값을 지어냄)** 를 막는 것이지, 출처가 분명한
선행 정보를 배제하라는 뜻이 아니다. 아래처럼 **사실**과 **가정**을 구분해 다룬다.

- **사실 (그대로 인용, 절대 날조 금지)**: DART 실적, KRX 시세, 그리고 **네이버 금융이 집계한
  증권사 컨센서스(선행 EPS·목표주가 평균·투자의견 평균)**. 컨센서스는 다수 증권사의 공개
  추정을 집계한 외부 출처값이며, 리포트에 출처·기준일을 명시한다.
- **가정 (규칙 기반·공개)**: 요구수익률 r, 장기성장률 g 시나리오, 컨센서스 밴드 폭 등.
  코드 상수로 두고 리포트에 전부 표기한다. LLM이 임의로 정하지 않는다.

밸류에이션 방법론:
- **RIM(잔여이익모델)** — 후행 실적 기반 내재가치 앵커: BPS × (ROE−g)/(r−g). 컨센서스가
  불필요하고 금융업 포함 범용. 사이클 저점에선 과소평가 경향이 있으므로 단독으로 쓰지 않는다.
- **CONSENSUS(증권사 컨센서스)** — 선행 신호: 네이버 금융 집계 목표주가 평균 ±스프레드 밴드.
  애널리스트 커버리지가 있을 때만. 후행 RIM의 저점 과소평가를 보정한다.
- **유형·커버리지별 가중치**: 일반(컨센서스 있음) RIM 0.35+컨센서스 0.65, 금융·지주 0.5+0.5,
  적자기업은 컨센서스 단독(있을 때만), 컨센서스 없으면 RIM 단독. 가중치는 리포트에 표기한다.
- v2의 "멀티플 = EPS × 현재 PER × (1+g)"는 목표주가가 현재가로 회귀하는 **순환 구조**라 폐기.
- 간이 DCF는 확장 단계 옵션 (현금흐름 안정 기업 한정)

### 5.3 목표주가 합성
- **목표주가 = 방법론별 결과의 가중평균, 보수/기본/낙관 3밴드로 제시**
- 투자의견은 **규칙 기반**: 기본 밴드 상승여력 +15% 이상 = 매수, −15% ~ +15% = 중립, −15% 미만 = 매도
- 산출에 쓴 방법론·가중치·가정(요구수익률, 성장률 시나리오)을 리포트에 반드시 표기

### 5.4 Peer 자동 선정 규칙
- KRX 업종분류 동일 업종 내에서 시가총액 유사도(대상 종목의 0.2~5배 범위) 상위 3~5개사 자동 선정
- 선정된 Peer 목록을 리포트에 표기 (적정 멀티플의 근거이므로 숨기지 않는다)

### 5.5 계산의 소유권
- 모든 계산은 **단위 테스트를 동반한 순수 함수**로 구현. LLM에 계산을 위임하지 않는다
- 테스트 기준 종목: 삼성전자(일반), 신한지주(금융), 카카오(성장주), 적자기업 1개 — 수작업 계산과 대조

## 6. LLM 리포트 생성 레이어

### 6.1 역할 분리 (v2 핵심 수정)
- **LLM 출력 스키마에 수치·의견 필드를 두지 않는다.** `opinion`, `target_price`, `upside_pct`를
  LLM이 재출력하게 하면 전사 과정에서 변형·창작 여지가 생긴다 — 이 값들은 백엔드 계산값을 그대로 쓴다
- LLM 출력은 **서술 필드 전용**:
  `summary[3]`(핵심 논거), `business`(사업 서술), `earnings_comment`(실적 해석),
  `valuation_comment`(산출 논리 서술), `strengths[3]`, `risks[3]`, `analyst_view`(종합)
- 최종 리포트 = 백엔드 계산값 + LLM 서술을 **코드에서 병합**

### 6.2 출력 형식 강제
- "JSON만 출력하라"는 프롬프트 강제 대신 **Claude API tool use로 스키마를 강제**한다 (구조 파싱 실패 원천 차단)

### 6.3 프롬프트 제약 (시스템 프롬프트에 상시 포함)
- "제공된 <데이터>에 없는 수치를 절대 만들지 말 것. 없으면 '자료 없음'으로 표기"
- "강점과 리스크를 동등한 비중으로. 매수 일변도 금지"
- "판단(목표주가·투자의견)은 제공된 계산값의 근거를 서술만 한다"

### 6.4 프로그램적 수치 검증 (프롬프트만으로는 부족)
- 생성된 서술에서 숫자를 정규식으로 추출 → **입력 JSON에 존재하는 값(및 단순 반올림 허용치)인지 코드로 대조**
- 불일치 발견 시: 해당 문장 지적과 함께 1회 재생성 → 재실패 시 해당 섹션 "검증 실패" 처리 (조용히 통과시키지 않는다)

### 6.5 캐싱
- 동일 종목 · 동일 분기 데이터 · 동일 프롬프트 버전이면 생성 결과 재사용

## 7. 단계별 개발 로드맵

**실행 전략: 수직 슬라이스 우선.** Phase 순서대로 수평 완성하는 게 아니라,
"삼성전자 1개 종목의 리포트가 end-to-end로 나오는 것"을 최우선으로 관통시킨다.
최대 리스크(DART 정합성, LLM 품질)를 3주 내 눈으로 확인하고, 이후 확장은 검증된 파이프라인의 반복이다.

### Phase 0 — 포맷 검증 + 준비 (2~3일)
- [ ] **코딩 전에**: claude.ai/Claude Code에 한국주식 MCP 연결 → "삼성전자 애널리스트 보고서" 생성 실험
      → **최고 출력물을 앱의 목표 스펙 문서로 확정** (이 계획의 §4.1 카드 구성과 대조·보정)
- [ ] OpenDART API 키 발급, `corpCode.xml` → 종목 매핑 마스터 구축
- [ ] Next.js + TypeScript + Tailwind 스캐폴딩, Vercel 배포 파이프라인, `.env` 구조
- ✅ 완료 기준: 목표 스펙 문서 1부 + 배포 URL에서 빈 페이지가 모바일로 열림

### Phase 1 — 데이터 레이어: 삼성전자 관통 (4~5일)
- [ ] DART 클라이언트: 기업개황 + 재무제표(연간 3~5년, 분기 4개) + 최근 공시 + 계정과목 정규화 매핑
- [ ] 시세 클라이언트: 현재가, 시총, 52주 밴드, 업종분류 (인터페이스로 추상화 — 공식 API 전환 대비)
- [ ] 캐시 레이어 (TTL 차등: 재무=분기, 시세=일, 마스터=월)
- ✅ 완료 기준: `/api/company/005930`이 정규화된 "리포트 원자재 JSON"을 반환

### Phase 2 — 분석 엔진 (4~5일)
- [ ] 재무비율 계산기 (성장성·수익성·안정성 핵심 12개 지표)
- [ ] 밸류에이션: 멀티플(시나리오 3밴드) + RIM, **업종별 분기 로직**(§5.1)
- [ ] Peer 자동 선정(§5.4), 목표주가 합성 + 투자의견 규칙(§5.3)
- [ ] 단위 테스트: 삼성전자 수치 수작업 대조
- ✅ 완료 기준: 테스트 통과 + 목표주가 3밴드가 산출 근거와 함께 JSON 출력

### Phase 3 — LLM 리포트 생성 (3~4일)

## 8. 법적 유의사항 (필수)

- 모든 화면·PDF에 면책 문구:
  > "본 리포트는 공시 데이터 기반 자동 생성 참고자료이며, 투자 권유가 아닙니다. 투자 판단의 책임은 이용자 본인에게 있습니다."
- **면책 문구만으로 충분하지 않을 수 있다**: 불특정 다수 대상 공개 서비스로 자동 목표주가·투자의견을
  제공하는 것은 자본시장법상 **유사투자자문업 신고 대상 여부 검토가 필요**하다.
  개인용·포트폴리오 시연용은 무관하나, 공개 배포 전 반드시 확인한다 (Phase 6 게이트)
- DART/KRX 데이터 이용약관 준수 (재배포 조건), 시세는 지연 시세임을 명시

## 9. 참고 — 하네스 엔지니어링(Harness Engineering) 핵심 지침

에이전트 기반 코딩(본 프로젝트의 Phase 3 LLM 파이프라인 설계에도 적용)에서 가장 널리 인용되는 지침:

1. **"가능한 가장 단순한 솔루션을 찾고, 필요할 때만 복잡성을 추가하라"** — Anthropic *Building Effective Agents*의 제1원칙이자 하네스 엔지니어링에서 가장 유명한 지침
2. **"하네스의 모든 구성요소는 '모델이 스스로 못 한다'는 가정을 인코딩한다 — 그 가정을 계속 스트레스 테스트하라"** — Anthropic *Harness design for long-running application development*
3. **Generator–Evaluator 루프**: 생성 에이전트와 비판적 평가 에이전트를 분리해 피드백 루프 구성

→ 본 프로젝트 적용: LLM에게 계산을 시키지 않고(가정: 모델의 산술은 신뢰 불가) 코드가 계산·LLM이
서술하는 단순한 단일 패스 구조로 시작한다. 단, 수치 검증기(§6.4)는 Evaluator 패턴의 최소 구현으로
처음부터 포함한다 — 금융 수치의 오류 비용이 크기 때문이다.
