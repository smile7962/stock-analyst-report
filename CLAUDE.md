# CLAUDE.md — 개발 에이전트 지침

> 본 파일은 이 저장소에서 코드를 작성하는 AI 코딩 에이전트(Claude Code 등)의 행동 지침이다.
> Andrej Karpathy가 정리한 LLM 코딩 실수 방지 4원칙에, 본 프로젝트(주식 애널리스트 리포트 웹앱)
> 전용 규칙을 더한 것이다. 제품 런타임의 리포트 생성 프롬프트와는 별개 레이어임에 유의한다.

---

## A. Karpathy 4원칙 (범용)

### 1. Think Before Coding — 조용히 가정하지 말 것
- 요구사항·API 스키마·데이터 형태가 불확실하면 **추측해서 코드를 짜지 말고 먼저 확인**한다.
- 확인할 수 없는 상황이면, 가정한 내용을 코드 주석이나 응답에 **명시**하고 진행한다.
- "아마 이럴 것이다"로 필드명·응답 구조를 지어내지 않는다.

### 2. Simplicity First — 과잉 설계 금지
- 지금 필요한 가장 단순한 구현에서 시작한다. 추상화·계층·옵션은 **실제로 필요해질 때** 추가한다.
- 쓰지 않는 범용 유틸, 미래 대비용 파라미터, 불필요한 디자인 패턴을 선제적으로 넣지 않는다.

### 3. Surgical Changes — 요청된 부분만 수정
- 작업과 무관한 코드·포맷·import 순서를 건드리지 않는다.
- 리팩터링이 필요해 보이면 **먼저 알리고** 별도로 진행한다. 한 변경에 여러 목적을 섞지 않는다.

### 4. Goal-Driven Execution — 구체적 성공 기준과 검증
- 작업 시작 전 "무엇이 되면 완료인가"를 구체적으로 정의한다.
- 코드를 실제로 실행/테스트해 동작을 **관찰**한 뒤 완료로 보고한다. 테스트가 실패하면 실패를 그대로 보고한다.

## B. 본 프로젝트 전용 규칙

### 5. 수치는 코드가, 서술은 LLM이 (가장 중요)
- **모든 재무 수치·비율·밸류에이션 계산은 테스트된 순수 함수(TypeScript 모듈)에서 수행**한다.
- LLM(Claude API)에는 계산을 위임하지 않는다. LLM은 계산된 수치를 **인용·해석·서술**만 한다.
- 리포트 생성 프롬프트에는 "제공된 데이터에 없는 수치를 만들어내지 말 것" 제약을 항상 포함한다.

### 6. 외부 데이터 스키마는 실제 응답으로 확인
- DART / KRX / KIS API의 필드명·응답 구조를 **추측하지 않는다**. 실제 샘플 응답(또는 공식 문서)으로 확인한다.
- 계정과목명(예: 매출액/영업이익) 등은 회사·보고서 유형별로 다를 수 있으므로 정규화 매핑을 명시적으로 둔다.
- 금융업·지주사 등 재무제표 구조가 다른 예외 케이스를 항상 고려한다.

### 7. 비밀정보와 API 한도
- API 키(DART/KRX/KIS/Claude)는 **서버 측에만** 두고 클라이언트 번들·저장소에 절대 커밋하지 않는다. `.env`는 `.gitignore`에 포함한다.
- DART 일일 한도(20,000건)를 의식해 캐시를 우선한다. 재무제표는 분기, 시세는 일 단위 TTL로 캐싱한다.

### 8. 금융 도메인 정확성
- 목표주가·투자의견은 **규칙 기반 로직**으로 산출하고(LLM이 최종 결정하지 않음), 산출 가정을 리포트에 표기한다.
- 모든 출력 화면·PDF에 면책 문구를 포함한다: 투자 권유가 아닌 참고자료임을 명시.
- 밸류에이션 모듈에는 알려진 종목 기준 단위 테스트를 반드시 동반한다.

### 9. 검증 습관
- 데이터 레이어·분석 엔진 변경 시 대표 종목(삼성전자=일반, 신한지주=금융, 카카오=성장주)으로 정합성을 확인한다.
- 프론트엔드 변경은 모바일 뷰포트 기준으로 확인한다.
